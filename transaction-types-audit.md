# Transaction-types audit

_Date: 2026-04-28. Companion to `decisions.md` and `treasury.md`._

## Why this exists

Over the last few months the `transactions.kind` enum grew from a handful to **15** values, driven by features (real estate, recurring payments, shipment accruals, PSD events, partner loans). Some kinds were promoted to top-level picker entries when they could have been a flag on an existing kind; others were added as user-pickable when they should be auto-generated by another module; one (`adjustment`) was already removed once because it was awkward (decisions.md, 2026-04-27).

This document audits every active kind, flags structural issues, and proposes concrete next steps. It also addresses two known coverage holes — Transfers between own accounts, and non-cash journal entries (depreciation, accruals, FX revaluation, write-offs).

**No code changes here.** Each recommendation is a candidate for its own follow-up session.

---

## A note on the "Synthesized Dr / Cr" column

The app does **not** maintain a double-entry general ledger. `/accounts` is a holdings registry (decisions.md, 2026-04-25), and the side-effects of a transaction are spread across three places:

- A **treasury movement** (or none, for accruals) — increments/decrements an account balance
- A **contact ledger** entry — increments/decrements an A/R or A/P balance per contact
- A **P&L bucket** classification (`REVENUE_KINDS` / `EXPENSE_KINDS` in [src/features/profit-loss/queries.ts:28](src/features/profit-loss/queries.ts))

To make near-duplicates **visible at a glance**, §1 below renders a synthesized Dr/Cr line as if a GL existed. It is not what the database stores — it is a tool for spotting kinds that are functionally indistinguishable. Where the synthesized line is identical to another kind's, the "Distinguishable by Dr/Cr alone?" column flags it.

## §2 — Structural flags

### (a) Identical / near-identical synthesized Dr/Cr

- **`expense` (paid_by=partner) vs `expense` (paid_by=business).** Same `kind`, same Dr Expense, but the Cr leg differs (Bank vs Partner-claim). The distinction is encoded in row shape — `partner_id` non-null **and** `from_account_id` null — and consumed by `allocatePartnerReimbursements`. Working as designed (decisions.md, 2026-04-24), but it is a smell: the row's meaning is not legible from `kind` alone. **Recommendation:** document, do not refactor.
- **`expense` vs `other_expense`.** Same Dr/Cr, same P&L bucket, same KDV treatment. The only difference is whether `expense_type_id` is set. This is a true near-duplicate. → **R1**.
- **`client_payment` (export) vs `client_payment` (real-estate).** Same `kind` and same Dr/Cr; disambiguated by `real_estate_deal_id` / `revenue_source` (decisions.md, 2026-04-27). The split decides whether the row hits P&L as revenue. Working as designed; the predicate (`real_estate_deal_id != null OR revenue_source='real_estate'`) is the single source of truth. **Recommendation:** document the predicate prominently in `decisions.md` so future devs do not add a new `real_estate_payment` kind. → **R3**.
- **`supplier_invoice`.** The synthesized Dr/Cr (`Dr Expense · Cr A/P`) misrepresents the row's actual P&L behaviour: it is **not** counted as expense today (the COGS side comes from `shipment_cogs` for goods, and other supplier invoices simply hold a payable until paid). The synthesized Dr line is misleading without the disclaimer at the top of this document — readers should not assume the row recognises an expense.

### (b) Distinction encodes a property that should be a field

- **`other_income` ⇄ a hypothetical `income` kind for primary revenue.** Today, primary revenue rides on `shipment_billing` (export) and the real-estate predicate on `client_payment`. The lone "other" prefix on `other_income` reveals the asymmetry: there is no general-purpose income kind. **Recommendation:** when a non-shipment, non-real-estate revenue stream appears, prefer adding a `revenue_category` field on a unified income kind rather than promoting another `*_income` kind. → **R2** (deferred).
- **`other_expense` is a category, not a kind.** The "other-ness" is the absence of an `expense_type_id`. → **R1**.
- **`partner_loan_in` / `partner_loan_out` × `is_loan` flag.** Two axes of meaning are encoded in the identifier: direction (in/out) is the kind, capital-vs-loan-vs-reimbursement is the flag plus row-shape. The current model works (decisions.md, 2026-04-27), but the "what is this row?" question requires reading two fields plus a FIFO-allocation rule. **Recommendation:** document the three flavours of `partner_loan_out` in `decisions.md` so the FIFO rule's role stays legible.

### (c) Could be auto-generated rather than user-initiated

Already auto-generated and confirmed correct: `shipment_billing`, `shipment_cogs`, `shipment_freight`, `profit_distribution`.

Still user-initiated, but candidates to move:

- **`tax_payment` with `kdv_period`.** The `/tax` reconciliation page already pre-fills the form via URL params (decisions.md, 2026-04-26). Closing the loop — letting "Mark filed" produce the row directly without rendering the form — would remove a click and eliminate the "did the user change anything I pre-filled?" ambiguity. → **R4**.
- **Recurring-payment occurrences.** Already auto-spawned on "Mark paid". The kind is whatever the template specifies (`expense`, `other_expense`, `supplier_payment`, or `tax_payment`). This means recurring payments are a **second auto-spawn surface** alongside shipment accruals and PSD events — worth recording in a list so future readers know where to look. → **R5** (documentation only).

### R1 — Collapse `other_expense` into `expense` with optional `expense_type_id`

**Target shape.** One `expense` kind. `expense_type_id` becomes nullable. Rows formerly typed `other_expense` migrate to `expense` with `expense_type_id = NULL`. The form distinguishes "categorised expense" vs "uncategorised expense" by whether the user picks a category — not by which kind they picked at step 1.

**Migration sketch.**
1. New SQL migration (`supabase/migrations/2026XXXX_collapse_other_expense.sql`):
   - `UPDATE public.transactions SET kind='expense' WHERE kind='other_expense';`
   - Update `transactions_kind_check` CHECK constraint to drop `other_expense` from the allowed set.
2. `src/lib/supabase/types.ts` — remove `other_expense` from `TRANSACTION_KINDS`.
3. `src/features/transactions/constants.ts` — remove from labels/descriptions/badges/`KIND_CATEGORIES`/`KIND_SPAWN_DIRECTION`/`CASH_OUT_KINDS`.
4. `src/features/transactions/schema.ts` — make `expense_type_id` optional on the `expense` schema; remove the `other_expense` schema entry.
5. `src/features/profit-loss/queries.ts` — drop `other_expense` from `EXPENSE_KINDS` (only `expense` needed).
6. KDV summary in `src/lib/ledger/kdv-summary.ts` — drop `other_expense` from VAT-bearing kind list (only `expense` needed).
7. Regenerate types: `npm run db:types`.

**Why this is safe.** Additive in the database (one UPDATE + a CHECK loosening followed by tightening). The kind is removed from the form picker — existing rows already migrated keep their behaviour because they now match `expense`. KDV and P&L bucketing are unaffected since the new union is the same.

### R2 — Mirror collapse for `other_income` _(deferred)_

Same shape as R1, but **don't act yet**. Today `other_income` is the only general-purpose income kind. Acting on it without first introducing a primary-income story (or a `revenue_category` field) would leave the system with no way to record income at all. Revisit when a non-shipment, non-real-estate revenue stream appears — a clear signal is the first time someone asks "where do I record consulting income?" or similar.

### R3 — Document the real-estate predicate

**Target shape.** A short, prominent block in `decisions.md` (or a new `revenue-recognition.md`) stating: "A `client_payment` is real-estate revenue iff `real_estate_deal_id IS NOT NULL OR revenue_source='real_estate'`. Export-side `client_payment`s are excluded from P&L to avoid double-counting against `shipment_billing`. The predicate lives in [profit-loss/queries.ts:54](src/features/profit-loss/queries.ts) — keep it there; do not duplicate."

**Why.** Decisions.md 2026-04-27 already records this, but the entry is buried. A future dev seeing two flavours of the same kind will be tempted to fork; the explicit "do not fork" rule prevents that.

**Effort.** XS — a few paragraphs.

### R4 — Auto-generate `tax_payment` from `/tax`

**Target shape.** Replace the "Record payment" link (which seeds the transaction form via URL params) with a "Mark this month filed" button that posts the row directly. Account, currency, and amount are already known from the KDV calculation; the only required user input is the source account.

**Migration sketch.**
1. New mutation `markKdvMonthFiled` in `src/features/tax/mutations.ts`. Takes `kdv_period`, `from_account_id`, `transaction_date`, `reference_number`. Writes a `tax_payment` row with `kdv_period` set.
2. Replace the "Record payment" link on `/tax` with a small inline form (account picker + date + reference). Keep an "advanced" affordance that opens the full transaction dialog if the user wants to override the auto-computed amount.
3. Add `tax_payment` to `DISABLED_KINDS`? — **No.** The kind remains user-pickable for non-KDV tax payments (income tax, withholding). Only the KDV-linked path is automated.

**Effort.** S. **Risk.** Low — additive UI; existing form path remains.

### R5 — Document auto-spawn surfaces

**Target shape.** A bullet list in `CLAUDE.md` or `decisions.md`:
> Transactions are auto-spawned by **three** modules: shipments (`shipment_billing/cogs/freight` at `draft → booked`), PSD events (`profit_distribution`, one event → N rows), and recurring payments (templated `expense/other_expense/supplier_payment/tax_payment` on "Mark paid"). When debugging "where did this row come from?", check these three sources before assuming user input.

**Effort.** XS.

### R6 — Add Transfer + Trade to step 1 of the wizard

**Target shape.** A new top-level option **"Move money"** appears in step 1 alongside the existing "Money in / Money out / Bills (accruals)". It fans out to two sub-options:

- **Transfer** (same asset) — same-currency move between own accounts (Kasa → bank, USD bank → USD bank).
- **Trade** (cross asset) — cross-currency conversion (USD → TRY, TRY → grams of gold).

Both write to `treasury_movements`, **not** to `transactions`. They reuse the existing treasury mutations and validation rules ([treasury/schema.ts:120](src/features/treasury/schema.ts) for transfer's same-asset check; the existing trade event for cross-asset).

**Wizard sketches.** See §4.

**Migration sketch.**
1. No DB changes required (treasury_movements already supports `kind='transfer'` and `kind='trade'`).
2. `src/features/transactions/constants.ts` — extend `KIND_CATEGORIES` with a fourth entry, `id: "move"`, that holds two synthetic sub-options. (These are **not** `TransactionKind` values; they are wizard-only branches.)
3. `src/features/transactions/transaction-form-dialog.tsx` — when a "move" branch is selected, render new step-2/step-3 panels that build a treasury-movement payload instead of a transaction payload. On submit, call the treasury mutation directly.
4. The transactions list does not change. The treasury-movements list (already exists at `/treasury`) continues to be the canonical view for transfers/trades.

**Effort.** M. **Risk.** Low — no schema change; the treasury side already works.

### R7 — Decide on the `order_billing` stub

**Current state.** Defined in `TRANSACTION_KINDS`, in `REVENUE_KINDS`, in `DISABLED_KINDS`, in `ACCRUAL_KINDS` — but **never spawned by anything**. The schema has a stub entry; there is no creation path.

**Options.**
- **Remove it** until you need it. Decisions.md 2026-04-27 introduced shipment-based billing; if the order-billing model is genuinely deferred, dropping the kind keeps the enum honest.
- **Keep it** as a placeholder if you have concrete plans to enable order-level billing (separately from shipment billing) in the next 1–2 quarters.

**Recommendation:** remove. The kind is dead code; reintroducing it later costs one migration. Keeping a kind in the enum that the application can never produce makes future readers wonder if there's a creation path they're missing.

**Effort.** S. **Risk.** Low — equivalent to R1's pattern.

### R8 — Adjustment / journal entry as a top-level kind

**Target shape.** A new top-level option **"Adjustment / journal entry"** in step 1. Multi-line debit/credit form. **Does not** write to `transactions`; instead writes to two new tables:

```
journal_entries
  id, entry_date, description, reference_number, attachment_path, created_at

journal_entry_lines
  id, entry_id (FK), line_no,
  category (enum: cash | ar | ap | revenue | expense | equity | tax | other),
  direction (enum: debit | credit),
  amount, currency,
  contact_id (nullable), expense_type_id (nullable),
  account_id (nullable), partner_id (nullable),
  memo
```

**P&L integration.** Lines with `category in ('revenue','expense')` flow into the existing P&L bucketing — `src/features/profit-loss/queries.ts` adds a third source alongside transactions and the real-estate predicate. Lines with `category in ('cash','ar','ap','equity','tax','other')` are balance-sheet-only and do not touch the P&L view.

**The cash rule.** A line with `category='cash'` is **bookkeeping-only**. It does **not** spawn a treasury movement. For real cash movements (deposit, withdraw, transfer, trade), use the appropriate transaction kind or treasury option. This rule prevents double-booking. The form must surface it loudly — e.g. a warning when a `cash` line is added: "This is a memo line. It will not change account balances. Use Money in / Money out / Move money for actual cash movements."

**Why a fixed coarse enum and not a full GL chart.** A full chart of accounts (`gl_accounts` table) is explicitly deferred (decisions.md, 2026-04-25) — the holdings registry should not be conflated with GL design. The 8-value enum (cash, ar, ap, revenue, expense, equity, tax, other) covers the actual use cases (depreciation, accruals, FX revaluation, write-offs, opening-balance offsets) without locking in a chart structure that may need to change later. If a real GL chart is added in the future, the enum becomes a coarse classification on top of GL accounts; existing journal entries migrate trivially.

**Validation.** Sum of debits must equal sum of credits **per currency** on each entry. The form validates live; the database enforces via a check constraint or trigger on `journal_entries`.

**Migration sketch.**
1. New SQL migration creating `journal_entries` and `journal_entry_lines` plus their foreign keys, the line-balance trigger, and indexes on `entry_date` and `category`.
2. `src/features/transactions/constants.ts` — extend `KIND_CATEGORIES` with a `"adjust"` branch (wizard-only, not a `TransactionKind`).
3. `src/features/transactions/transaction-form-dialog.tsx` — branch into the new multi-line editor at step 2; submit writes via a new mutation.
4. New module `src/features/journal-entries/` — schema (Zod), mutations, queries, list view.
5. P&L queries — union journal-entry revenue/expense lines into `pAndLRows`.
6. KDV summary — extend to include `tax`-category lines if KDV is recorded via a journal entry.
7. Regenerate types.

**Effort.** L. **Risk.** Medium — touches P&L (a critical view). Mitigation: add the new lines to P&L behind a feature toggle initially; verify on a dummy entry before enabling for real bookkeeping.

## §4 — Coverage gap: Transfers, Trades, Adjustments

### Status today

| Capability | Exists? | How |
|-----------|---------|-----|
| Transfer (own accounts, same asset) | **Yes**, but hidden | `treasury_movements.kind = 'transfer'` — paired withdraw + deposit linked by `group_id`, same-asset enforced ([treasury/schema.ts:120](src/features/treasury/schema.ts)). Only reachable from `/treasury`. |
| Trade (own accounts, cross asset) | **Yes**, but hidden | `treasury_movements.kind = 'trade'`. Only reachable from `/treasury`. |
| Adjustment — quantity correction (single account) | **Yes** | `treasury_movements.kind = 'adjustment'` — signed-quantity correction for cash/inventory miscounts. |
| Adjustment — non-cash journal entry (multi-line Dr/Cr) | **No** | Removed from `transactions` on 2026-04-27 ([20260427210000_drop_transaction_adjustment_kind.sql](supabase/migrations/20260427210000_drop_transaction_adjustment_kind.sql)). Explicitly deferred in decisions.md as "replace with a true journal-entry feature". |

The two real gaps from the user's perspective are:

1. **Discoverability of Transfer / Trade.** A user thinking "I want to log a transaction" reasonably reaches for the new-transaction wizard. They do not find Transfer there; they would have to know about `/treasury`. → R6.
2. **Non-cash journal entries.** Genuinely missing. Workarounds today (`other_expense` with a memo) lose information — they spawn a fake cash movement and cannot represent multi-line entries. → R8.
